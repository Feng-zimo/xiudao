# 🌿 网页修仙模拟器 - 需求理解与实现计划（已更新）

## 📋 一、项目核心理解

### 1.1 项目定位
- **纯前端文字修仙游戏**：完全运行在浏览器中，无需后端服务器
- **零成本部署**：GitHub Pages 托管，打开即玩
- **单机体验**：本地存储，离线可玩
- **社交元素**：通过口令交换存档，实现"道友交流"

### 1.2 核心特色
✅ 无注册、无联网、无付费  
✅ 口令存档系统（Base64编码 + 密码保护）  
✅ 多职业、多阵营、多天赋系统  
✅ 完整的修仙境界体系  
✅ 风险与代价并存的修炼机制  

---

## 🏗️ 二、技术架构设计

### 2.1 技术栈
```
前端：HTML5 + CSS3 + Vanilla JavaScript (ES6+)
存储：localStorage（浏览器本地存储）
加密：Base64编码 + Hash校验（自动选择最优算法）
部署：GitHub Pages
```

### 3.2 修为境界体系

#### 境界划分
```
练气期 (Lv1-10) → 筑基期 (Lv1-10) → 结丹期 (Lv1-10) 
→ 金丹期 (Lv1-10) → 元婴期 (Lv1-10) → 筠仙期 (Lv1-10)
```

#### 升级机制
- **经验获取**：击杀怪物、服用丹药
- **突破条件**：
  - 练气→筑基：积累足够灵力
  - 筑基→结丹：**需要渡劫**（风险机制）
  - 结丹→金丹：高阶瓶颈
  - 金丹→元婴：极难突破
  - 元婴→筠仙：传说境界

#### 邪修特殊规则
- ❌ 无法直接结婴
- ✅ 结丹圆满后可开启**第二人格**
- ✅ 可通过**灵魂夺舍**获得怪物肉体
- ⚠️ 夺舍不完全会导致修为大跌

### 3.4 战斗系统

#### 战斗模式：**回合制**

**战斗流程**：
```
1. 玩家选择攻击方式（普通/技能/控云法术）
2. 计算伤害 = 攻击力 - 防御力 + 法术加成
3. 判定暴击（基于幸运值）
4. 敌人受到 damage
5. 敌人回合，同样逻辑
6. 循环直到一方HP<=0
```

**胜利奖励**：
- 经验值（用于提升境界）
- 掉落物（装备、材料、丹药）
- 邪修额外获得魂魄

**失败惩罚**：
- 修为损失（建议5-10%）
- 可能重伤（需要疗伤丹药）

### 3.5 风险与代价系统（重点）

| 行为 | 风险 | 后果 |
|------|------|------|
| 灵兽过强 | 反噬、叛逃 | 修为损失、灵兽离开 |
| 服用过猛丹药 | 炸修为/炸肉体 | 境界跌落、重伤 |
| 第二人格融合 | 第一人格破碎 | 角色重置或部分属性丢失 |
| 夺舍不完全 | 修为大跌 | 境界大幅跌落 |
| 交换存档 | 可能遭人暗算 😂 | 存档被篡改、角色死亡 |

**设计理念**：天道无情，但讲道理。高风险高回报。

### 3.5 存档与口令系统（核心重点）

#### 3.5.1 本地存档
```javascript
// 使用 localStorage
localStorage.setItem('xiudao_save', JSON.stringify(gameData));
```
- ✅ 关闭网页不丢失
- ✅ 自动保存（关键节点触发）
- ✅ 手动保存按钮

#### 3.5.2 口令导出系统

**数据结构设计**：
```javascript
{
  // 游戏数据
  player: {
    name: "张三",
    faction: "正道", // 或 "邪修"
    profession: "召唤师",
    talent: "运气",
    level: {
      realm: "结丹期",
      stage: 5  // 当前境界等级
    },
    attributes: {
      hp: 1000,
      mp: 500,
      attack: 100,
      defense: 50
    },
    experience: 10000,
    // ... 更多属性
  },
  
  pets: [
    {
      name: "小青",
      type: "青龙",
      level: 8,
      loyalty: 80,
      // ...
    }
  ],
  
  inventory: [
    {
      id: "item_001",
      name: "聚灵丹",
      count: 10,
      // ...
    }
  ],
  
  // 邪修专属
  evilCultivator: {
    souls: 100,
    secondPersonality: null, // 或对象
    possessionTarget: null,
    refiningFlesh: {
      target: "妖兽王",
      daysRemaining: 30,
      medicine: "九转还魂丹"
    }
  },
  
  // 控云法术
  cloudControl: {
    level: 5,
    element: "雷", // 风/雨/雷/电/火/冰
    damage: 150
  },

  // 元数据
  metadata: {
    createTime: "2024-01-01 12:00:00",
    lastSaveTime: "2024-01-15 18:30:00",
    playTime: 3600, // 秒
    version: "1.0.0"
  }
}
```

**加密流程**：
```
1. 将游戏数据转为 JSON 字符串
2. 添加用户密码（如果有）
3. 计算 Hash 校验值（防止篡改）
4. Base64 编码
5. 生成口令字符串
```

**口令格式设计**：
```
XIUDAO|v1.0|{Base64数据}|{Hash校验}
```

示例：
```
XIUDAO|v1.0|eyJwbGF5ZXIiOnsibmFtZSI6IuW4gj...|a3f5b8c9d2e1...
```

#### 3.5.3 密码保护机制

**四种情况处理**：

| 情况 | 处理方式 |
|------|----------|
| 无密码 | 仅可预览基本信息（境界、职业、灵兽数量、是否夺舍） |
| 密码错误 | 禁止导入，提示错误 |
| 数据被篡改 | Hash校验失败，拒绝导入 |
| 密码正确 | 正常载入存档 |

**预览模式显示内容**：
- ✅ 境界（如：结丹期 Lv5）
- ✅ 职业（如：召唤师）
- ✅ 灵兽数量（如：3只）
- ✅ 是否夺舍（是/否）
- ❌ 不显示详细属性、物品列表等

**Hash校验算法**（待确认）：
- 方案A：SHA256（更安全，但JS原生不支持，需引入库）
- 方案B：MD5（较快，有轻量级JS实现）
- 方案C：简单CRC32（最轻，但安全性低）

**推荐**：使用 SHA256，平衡安全性和性能。

#### 3.5.4 导入导出流程

**导出流程**：
```
1. 用户点击"导出存档"
2. 输入密码（可选）
3. 系统生成口令字符串
4. 显示在文本框中
5. 用户可复制分享给道友
```

**导入流程**：
```
1. 用户点击"导入存档"
2. 粘贴口令字符串
3. 如果存档有密码，要求输入
4. 验证Hash校验
5. 解密并加载数据
6. ⚠️ 覆盖当前存档（需二次确认）
```
